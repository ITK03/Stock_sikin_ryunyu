# 統合株式ダッシュボード (Stock_sikin_ryunyu)

日本株・米国株を対象に、**① 資金流入ランキング(自リポジトリ生成)・② セクター騰落率・③ 適時開示**の
3系統のデータを、同一オーナー(ITK03)の他リポジトリ(`Stock_open_news` / `sector-monitor`)から
統合して見られるスマホ最優先の静的サイト。銘柄コードをキーに3系統を横断連携し、
どの画面からでも銘柄コードをタップすればその銘柄の全体像(現在値・所属セクター・
ランキング順位・最新開示)を1画面で確認できる。

## 3つのタブ

- **資金流入**(このリポジトリが生成する既存機能。詳細は後述)
- **セクター** — `sector-monitor` が生成するセクター/テーマ別騰落率。JP/US切替、騰落率順表示、
  タップで構成銘柄(Tierバッジ・騰落率)を展開。
- **開示** — `Stock_open_news`(TDnet適時開示)が生成する最新開示。新しい順、重要度しきい値フィルタ、
  銘柄コード検索、方向(好材料/悪材料/中立)で色分け。

画面下部(モバイルは固定)のタブバーで切り替える。各タブの銘柄コード・構成銘柄・開示の
証券コードはすべてタップ可能で、**銘柄詳細シート**が開き、3データソースを横断した情報
(現在値・騰落率・所属セクター/テーマ・資金流入ランキングでの順位・最新開示一覧)を1画面に集約する。
データが無い項目は「データなし」と明示する。

## データソースと更新頻度

| # | データ | 生成元 | 取得方法 | 更新頻度 |
| --- | --- | --- | --- | --- |
| ① | 資金流入ランキング | このリポジトリ(下記) | ビルド時に `public/data/*.json` を生成 | 平日、場中〜引け後に複数回(後述) |
| ② | セクター騰落率 | [sector-monitor](https://github.com/ITK03/sector-monitor) | ブラウザから直接 fetch(専用ブランチ `data`) | sector-monitor 側の生成頻度に依存 |
| ③ | 適時開示(TDnet) | [Stock_open_news](https://github.com/ITK03/Stock_open_news) | ブラウザから直接 fetch(GitHub Pages → raw の順にフォールバック) | Stock_open_news 側の生成頻度に依存(継続的に更新) |

②③は他リポジトリが公開する JSON をクライアント側で直接 fetch する(このリポジトリのビルドは
関与しない)。候補URLを順に試し、全滅した場合はそのタブ(または銘柄詳細の該当セクション)に
エラー/未取得表示を出す(アプリ全体は落ちない)。

### ②セクターのデータ契約(schema_version 2)

sector-monitor 側の実測でデータが数MB規模になったため、地域ごとにファイルを分割し、
銘柄横断インデックスも独立ファイルにしている(専用ブランチ `data` のルート直下、
`raw.githubusercontent.com` から直接取得。旧 `itk03.github.io` 系のセクターURLは廃止)。

- `sector_jp.json` / `sector_us.json` — 地域ごとのセクター(テーマ)一覧。`members` は各セクター
  **上位30件のみ**(`count` が全構成数)。`ticker`/`price` フィールドは無い。
- `ticker_index.json` — 銘柄コード→所属セクターの逆引き。**日本株のみ**・所属セクターは全件
  (`sector_jp.json` の上位30件制限を受けない)。フィールドは `n`(名称)/`c`(騰落率)/`p`(現在値)/
  `s`(`[セクター名, Tier]` の配列)と短縮キー。

パフォーマンス上の理由から:
- JP/USはセクタータブで選択した市場だけを取得する(タブ切替時に初めて遅延fetch。初期表示では
  両方読み込まない)。JPは1500超のテーマになり得るため、セクタータブは「上位40件+もっと見る」の
  段階表示と、セクター名での検索フィルタでモバイルでも軽く動作するようにしている。
- `ticker_index.json` は銘柄詳細シートを最初に開いたときに遅延fetchする(初期ロードに含めない)。
  一度取得すればモジュールスコープのメモリキャッシュを再利用するため、タブ間・複数回の詳細表示で
  重複フェッチしない。
- これら数MB規模のファイルは `sessionStorage` には保存しない(容量超過で失敗しうるため)。
  メモリキャッシュ + ブラウザのHTTPキャッシュに任せる。③の開示(数百件程度、小さい)のみ
  数分間 `sessionStorage` にキャッシュし、タブ切替のたびの再取得を抑える。

米国株は `ticker_index.json` の対象外のため、銘柄詳細では `sector_us.json` の構成銘柄一覧から
ベストエフォートで所属セクター/騰落率を拾う(現在値は含まれないため「データなし」になる)。

### オフライン開発用のサンプルモード

②③は他リポジトリの本番データに依存するため、ネットワークが無い環境でも動作確認できるよう、
URLに `?sample=1` を付けるとバンドル済みのサンプルデータ(`src/data/sampleSector.ts` /
`src/data/sampleDisclosures.ts`)を使う「サンプルモード」で全タブを確認できる
(実際の外部 fetch は行わない)。①のランキングは元々 `npm run build:data:sample` で
同様の決定論的サンプルを生成できる(後述、既存機構のまま)。サンプルモードでは
セクター/開示タブに「サンプル」バッジを表示する。

## 資金流入ランキング(① 既存機能)

日本株のうち「時価総額に対して売買代金が大きい」=資金流入が強い銘柄を見つける機能。

### 3つのランキング

- **① 時価総額比** — 最新営業日のスナップショット。`売買代金 / 時価総額` が大きいほど上位。
- **② 連日継続** — 選択した期間(3日 / 1週間 / 2週間 / 1ヶ月 / 3ヶ月 / 半年)の**平均**で、時価総額比の売買代金が大きいほど上位。資金流入が連日続いている銘柄が浮上する。
- **③ 全市場上位** — ②に加えて、**全市場の売買代金(期間平均)上位 topK 位以内**に入る銘柄だけに絞り込み。各行に全市場での売買代金順位を表示。
- **④ 急増(初動)** — 直近1〜3日平均の売買代金が、過去25営業日平均(平常時)に対して何倍かで降順。

指標の定義: `比率 = 売買代金 / 時価総額`(時価総額 = 終値 × 発行済株式数)。

### 実データ(無料 / Yahoo Finance + JPX)

**APIキー・課金・登録は不要。** 2つの無料ソースだけを使う:

- **ユニバース**: JPX公式「東証上場銘柄一覧(data_j.xls)」→ 全市場の内国株式(プライム/スタンダード/グロース)+ 市場区分
- **株価・出来高の履歴**: Yahoo Finance chart API(認証不要、6ヶ月〜1年の日足)
- **発行済株式数(時価総額用)**: Yahoo Finance quote API

```bash
npm run universe     # JPX一覧+Yahoo株式数で config/universe.json を生成
npm run build:data   # Yahoo から取得 → ランキング計算 → JSON 出力
npm run build:web
```

### 指標の算出(無料データの制約)

| 指標 | 算出 |
| --- | --- |
| 売買代金 | `終値 × 出来高`(出来高代金の近似) |
| 時価総額 | `終値 × 発行済株式数` |
| 比率 | `売買代金 / 時価総額` |

> 真の売買代金(Σ価格×数量)は無料では銘柄横断・履歴で取得できないため、強く相関する
> `終値×出来高` を用いる。ランキング用途では実用上問題ない。より厳密にしたい場合は
> `src/data/yahoo.ts` の `barsFromChart` を別ソースに差し替えればよい(プロバイダは
> `src/data/provider.ts` の `DataProvider` インターフェースで分離済み)。

### 鮮度

Yahoo の日足は当日分も場中〜引け後に更新される(おおよそ15〜20分遅延)。
ワークフローは平日に**場中(毎時)・寄り直後・引け直前・引け後**の複数回走るので、
当日の値を当日中に反映できる(1営業日以上の遅延なし)。

> 注: 開発サンドボックスは外部ドメインがegress制限で遮断されるため、実取得は
> ネットワーク開放の **GitHub Actions 上**で実行される。ローカルで実データを試す場合は
> ネットワーク制限のない環境で `npm run universe && npm run build:data` を実行する。
> ②③(セクター・開示)の外部fetchも同様の理由でサンドボックスからは到達できないことがある
> (その場合はアプリ側がエラー表示にフォールバックする。`?sample=1` で動作確認可能)。

## スマホから見る(GitHub Pages)

1. リポジトリの **Settings → Pages → Build and deployment → Source** を **GitHub Actions** に設定。
2. `.github/workflows/deploy.yml` が**平日に複数回**(場中・引け前・引け後)自動で
   ①を再生成しデプロイ(②③はビルド不要、閲覧時にブラウザが直接取得)。手動実行
   (Actions タブ → Run workflow)や push でも走る。
3. 公開URL(`https://<user>.github.io/<repo>/`)をスマホのホーム画面に追加。

シークレットや課金は不要。ローカル確認用のサンプルデータも同梱されている。

## 設計上のパラメータ

`src/pipeline/build.ts` で調整可能:

- `topN` (既定100) — 各リストの表示件数
- `topK` (既定100) — ③で「全市場上位」とみなす売買代金順位の閾値
- `minCoverage` (既定0.6) — 期間内に必要なデータ被覆率(欠損の多い銘柄を除外)
- `LOOKBACK_DAYS` (既定130) — 取得する営業日数(半年=約120営業日 + 余裕)
